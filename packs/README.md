# Presentation packs

Client-side cosmetic packs (art + palette). A pack **never** affects rules — it
only changes what your client draws — so it needs no server authority, no content
hash, and no validation tier (see `ARCHITECTURE.md` §6).

Point the game at a pack directory with the `ATB_PACK` environment variable:

```bash
ATB_PACK=packs/default ./build-gui/tactical_battler
```

Absent or unloadable ⇒ the built-in primitive renderer (identical to no pack).

Two packs ship in this repo:

- **`default/`** — a full sprite pack (tiles, unit tokens, and a distinct icon
  per spell) generated by `tools/gen_pack.py` — a zero-dependency, pure-Python
  atlas generator (draws `atlas.png` + emits a matching `pack.json`). Regenerate
  or fork it with `python3 tools/gen_pack.py [name]`.
- **`example_upscaled_to_96px/`** — the same look at a higher resolution (96 px
  cells instead of 48), handy as a starting point for crisper art.

Both champions have a little **cast animation** in these packs — watch a unit
when it casts (see *Animations* below).

Packs only theme the **battle** screen — hit *Fight* to see it (the build editor
isn't pack-routed yet). Watch the console for a `Loaded sprite pack …` line on
startup.

## Format

A pack is a directory with a `pack.json` manifest, plus any atlas PNGs it names.
Every key is optional and resolves independently, so a pack can restyle just the
walls, or ship **only a palette and no art at all** (like `example/` here).

```jsonc
{
  "schema": 1,
  "name": "My Pack", "version": "1.0.0",
  "tileSize": 32,
  "atlases": { "main": "atlas.png" },       // optional: sprite sheets
  "palette": { "wall": "#464e60" },          // optional: recolour primitives
  "sprites": {                               // optional: atlas sub-rects
    "tiles.wall":       { "atlas": "main", "rect": [0, 0, 32, 32] },
    "units.player":     { "atlas": "main", "rect": [0, 64, 32, 48], "anchor": "bottom" },
    "spells.fireball":  { "atlas": "main", "rect": [0, 96, 32, 32] }
  }
}
```

### Keys the renderer looks up (v1)

| Element          | Sprite key            | Palette key            |
|------------------|-----------------------|------------------------|
| Floor tile       | `tiles.floor`         | `floor`                |
| Wall tile        | `tiles.wall`          | `wall`                 |
| Obstacle tile    | `tiles.obstacle`      | `obstacle`             |
| Player champion  | `units.player`        | *(faction tint)*       |
| Enemy champion   | `units.enemy`         | *(faction tint)*       |
| Summon / object  | `units.<name>` then `spells.<name>` (e.g. `units.brute` → `spells.brute`) | — |
| Spell icon (bar) | `spells.<catalog key>` (e.g. `spells.fireball`) | — |

Resolution ladder per element: **atlas sprite → palette colour → built-in
primitive**. For a **summoned creature or object** (bomb, blocker, healer, brute)
the renderer tries `units.<name>` first, then falls back to its **spell icon**
`spells.<name>`, then to the faction sprite — so a pack that already draws the
spell icons gets on-field creature art for free (add a dedicated `units.<name>`
only if you want the field look to differ from the bar icon). `anchor` is
`center` (default) or `bottom`.

Ground effects and status markers aren't pack-routed yet — that wiring is a
follow-up; the seam already supports the same key→sprite lookup.

## Editing the art (for artists)

You don't need to touch any code. A pack is just:

1. **`atlas.png`** — one image holding all the little pictures, laid out on a
   grid of equal squares (48×48 in `default/`). Each square is one sprite: a
   floor tile, a champion, a spell icon, and so on.
2. **`pack.json`** — a text file that tells the game *where* each picture lives
   in that image.

To **re-skin the game**, open `atlas.png` in any image editor and repaint the
squares — keep them the same size and in the same spots and you're done; nothing
in `pack.json` needs to change. If you'd rather rearrange or add pictures, each
line in `pack.json` points at a rectangle:

```jsonc
"units.player": { "atlas": "main", "rect": [144, 0, 48, 48] }
//                                          └x   └y └w  └h
```

`rect` is `[x, y, width, height]` in **pixels**, where `x, y` is the top-left
corner measured from the top-left of the image. So `[144, 0, 48, 48]` means "the
48×48 square starting 144 pixels from the left, at the very top." Move a picture
in the atlas → update those four numbers to match.

## Animations

An animation is nothing more than **a few of these rectangles shown one after
another**, like a flip-book. You paint each frame as its own square in the atlas,
then list them. There are two kinds, and a sprite can have either or both:

- **`anim`** — an *idle* loop that plays forever (e.g. a torch flickering).
- **`cast`** — a *one-shot* that plays once when that unit casts a spell, then
  snaps back to its normal picture.

```jsonc
"units.player": {
  "atlas": "main",
  "rect": [144, 0, 48, 48],                 // the normal, resting picture
  "cast": {
    "rects": [ [240, 0, 48, 48],            // frame 1: a burst
               [240, 0, 48, 48],            // frame 2: hold the burst a beat
               [144, 0, 48, 48] ],          // frame 3: back to normal
    "fps": 12                               // frames per second (bigger = faster)
  }
}
```

- `rects` is the list of frames, in order — each one a `[x, y, width, height]`
  rectangle just like `rect`.
- `fps` sets the speed. `loop` is optional: `anim` loops by default, `cast`
  plays once by default — set `"loop": true`/`false` to override.
- Frames can reuse squares that already exist in the atlas (the shipped cast
  flash just borrows the attack-burst square), so you can add motion **without
  drawing anything new**.

The two shipped packs use exactly this to make champions flash when they cast —
run one and cast a spell to see it. If a pack leaves `anim`/`cast` off, the
sprite simply stays still, so animations are always optional.
