# Authoring a sprite pack

This is the full guide to making a **presentation pack** for Tactical Battler — a
folder of art (and/or a palette) that changes how your client draws the game. It
covers three ways in, easiest first:

1. **Recolour the built-in look** with a palette-only pack — no art, no image
   editor, ~5 lines of JSON.
2. **Repaint the shipped atlas** — keep the layout, swap the pixels.
3. **Author your own atlas** — hand-draw rects, or export from a texture packer.

A pack is **100% client-side**: it never touches the rules, needs no server
authority, no content hash, and no validation tier (see `ARCHITECTURE.md` §6).
Two players in the same match can run entirely different packs — or none — and the
outcome is identical. Absent or unloadable ⇒ the game draws its built-in
primitives, exactly as if no pack were set.

Point the game at a pack with the `ATB_PACK` environment variable:

```bash
ATB_PACK=packs/example ./build-gui/tactical_battler
```

Watch the console for a `Loaded sprite pack …` line on startup. Packs currently
theme the **battle** screen only — hit *Fight* to see them (the build editor
isn't pack-routed yet).

## What ships in this repo

| Pack | What it is | Good starting point for… |
|------|------------|--------------------------|
| `packs/example/` | **Palette-only** — recolours the three tile types, no atlas | the fastest possible re-theme |
| `packs/default/` | **Full atlas** — tiles, unit tokens, a distinct icon per spell, cast animations | forking a complete pack |
| `packs/example_upscaled_to_96px/` | The `default` look at 96 px cells instead of 48 | crisper / higher-res art |

`packs/default/` and its upscaled twin are generated by
[`tools/gen_pack.py`](../tools/gen_pack.py) — a zero-dependency, pure-Python atlas
generator (it draws `atlas.png` **and** emits a matching `pack.json`, so the two
can't drift). Regenerate or fork one with:

```bash
python3 tools/gen_pack.py mypack     # → packs/mypack/{atlas.png,pack.json}
```

## 1. Palette-only — the fastest re-theme

The simplest possible pack is a `pack.json` with just a `palette`. It recolours
the built-in primitive tiles and ships **no image at all**. Copy `packs/example/`
and edit the three colours:

```jsonc
{
  "schema": 1,
  "name": "My Re-theme", "version": "1.0.0",
  "tileSize": 32,
  "palette": {
    "floor":    "#20303a",
    "wall":     "#3f6b5a",
    "obstacle": "#8a5a2b"
  }
}
```

Colours are `"#RRGGBB"` or `"#RRGGBBAA"`. The renderer looks up exactly three
palette keys today — `floor`, `wall`, `obstacle`. Everything else (units, spell
icons) keeps its built-in look, which is the whole point: **a partial pack
restyles only what it defines and everything else falls back cleanly.**

## 2. Repaint the shipped atlas

If you want custom pictures but not a custom layout, copy `packs/default/`, open
its `atlas.png` in any image editor, and repaint the squares. As long as you keep
each sprite the **same size** and **in the same spot**, nothing in `pack.json`
needs to change — the rects still point at the right squares.

`default/`'s atlas is a grid of 48×48 cells: tiles on the top row, the two
champions next to them, then one cell per spell icon. Repaint in place, save, run
with `ATB_PACK=packs/default` and you'll see your art.

## 3. Author your own atlas

A pack is a directory holding a `pack.json` manifest plus any atlas PNGs it names:

```
mypack/
  pack.json     ← the manifest: metadata + atlas list + key → rect map + palette
  atlas.png     ← one image holding every sprite (a pack may have several pages)
```

### Why one atlas instead of many PNGs

Raylib draws sub-rectangles of a *bound* texture, so the canonical format is a
**texture atlas** — one image, many sprites — not hundreds of loose files (each
would force a GPU bind per sprite, the slow path). One atlas = one bound texture:
sprites batch into far fewer draw calls, animations are just sub-rects advancing
on the same page, and there's one decode/upload instead of hundreds of file opens
(which also shrinks future WASM web builds — fewer files to preload).

### The manifest

Every block below `schema`/`tileSize` is **optional** and resolves independently,
so a pack can restyle just the walls, ship only a palette, or define every sprite.

```jsonc
{
  "schema": 1,
  "name": "Pixel Dungeon", "version": "1.0.0",
  "tileSize": 32,                            // authoring cell size (informational)
  "atlases": { "main": "atlas.png" },        // name → filename (relative to the pack dir)
  "palette": { "wall": "#464e60" },          // optional: recolour any primitive we still fall back to
  "sprites": {
    "tiles.wall":      { "atlas": "main", "rect": [0,  0, 32, 32] },
    "units.player":    { "atlas": "main", "rect": [0, 64, 32, 48], "anchor": "bottom" },
    "spells.fireball": { "atlas": "main", "rect": [0, 96, 32, 32] }
  }
}
```

- `rect` is `[x, y, width, height]` in **pixels**, measured from the top-left of
  the named atlas image. `[144, 0, 48, 48]` = the 48×48 square 144 px from the
  left, at the very top.
- `anchor` is `center` (default) or `bottom` — use `bottom` to stand a tall
  sprite on its tile instead of centring it.
- Unknown keys and malformed rects are **rejected** with a contextual error
  (e.g. `sprites.tiles.wall: "rect" width/height must be > 0`); a bad pack fails
  to load rather than drawing garbage.

### The keys the renderer looks up (v1)

| Element | Sprite key | Palette key |
|---------|------------|-------------|
| Floor tile | `tiles.floor` | `floor` |
| Wall tile | `tiles.wall` | `wall` |
| Obstacle tile | `tiles.obstacle` | `obstacle` |
| Player champion | `units.player` | *(faction tint)* |
| Enemy champion | `units.enemy` | *(faction tint)* |
| Summon / object | `units.<name>` then `spells.<name>` (e.g. `units.brute` → `spells.brute`) | — |
| Spell icon (bar) | `spells.<catalog key>` (e.g. `spells.fireball`) | — |
| Shelter wall | `surfaces.wall` | *(built-in block)* |
| Elemental surface | `surfaces.<element>` — `fire`/`water`/`ice`/`poison`/`electric`/`heal`/`oil`/`steam` (neutral glyph = `surfaces.glyph`) | *(element tint)* |

Resolution ladder per element: **atlas sprite → palette colour → built-in
primitive**. For a summoned creature or object (bomb, blocker, healer, brute) the
renderer tries `units.<name>` first, then its **spell icon** `spells.<name>`, then
the faction sprite — so a pack that already draws the spell icons gets on-field
creature art for free. Add a dedicated `units.<name>` only if you want the field
look to differ from the bar icon.

> Ground effects (walls + elemental surfaces) are pack-routed via the
> `surfaces.*` keys above — a surface sprite is layered over the floor tile, or
> falls back to the flat element tint. Status markers aren't pack-routed yet.

## Exporting an atlas from a texture packer

You don't have to place sprites by hand. Any tool that produces a packed sheet
works — only the final `[x, y, w, h]` rects matter:

- **[free-tex-packer](https://free-tex-packer.com/)** — free, in-browser or
  desktop. Drop your loose PNGs in, and it emits `atlas.png` plus a data file with
  each sprite's frame rectangle. Read the `frame: {x, y, w, h}` for each sprite and
  copy those four numbers into the matching `rect` in `pack.json`.
- **[TexturePacker](https://www.codeandweb.com/texturepacker)** — the commercial
  standard. Export any of its generic/JSON data formats; each entry's
  `frame` (`x, y, w, h`) maps straight to a `pack.json` `rect`.

Neither tool knows Tactical Battler's key names, so the workflow is: pack your
sprites → note each frame's rect → author `pack.json` mapping our semantic keys
(`tiles.wall`, `spells.fireball`, …) to those rects. Keep the exported `atlas.png`
next to `pack.json` and list it under `atlases`.

> Loose per-file PNGs remain a fine *authoring* convenience — but the on-disk and
> in-VRAM form the game loads is the atlas.

## Animations

An animation is just **a few rects shown one after another**, like a flip-book.
You paint each frame as its own square in the atlas, then list them. There are two
kinds, and a sprite can have either or both:

- **`anim`** — an *idle* loop that plays forever (a torch flickering). Loops by
  default.
- **`cast`** — a *one-shot* that plays once when that unit casts a spell, then
  snaps back to its resting picture. Plays once by default.

```jsonc
"units.player": {
  "atlas": "main",
  "rect": [144, 0, 48, 48],                 // the normal, resting picture
  "cast": {
    "rects": [ [240, 0, 48, 48],            // frame 1: a burst
               [240, 0, 48, 48],            // frame 2: hold the burst a beat
               [144, 0, 48, 48] ],          // frame 3: back to normal
    "fps": 12                               // frames per second (bigger = faster)
  }
}
```

- `rects` is the ordered frame list — each a `[x, y, w, h]` rectangle like `rect`.
- `fps` sets the speed. `loop` is optional (`anim` loops, `cast` doesn't; set
  `"loop": true`/`false` to override).
- Frames can **reuse squares that already exist** in the atlas — the shipped cast
  flash just borrows the attack-burst cell, so you can add motion without drawing
  anything new.

The event trigger rides the engine's combat event stream: when the engine reports
a `Cast`, the frontend's `Animator` plays that unit's `cast` clip once. Because
clips are sub-rects of the already-bound atlas, animation costs no extra texture
loads. The two shipped atlas packs use exactly this to make champions flash when
they cast — run one and cast a spell to watch it. `hit`/`death` event clips are a
pure-data follow-up once packs supply the frames.

## Checklist

- [ ] Folder with a `pack.json`; any atlas PNGs sit next to it and are named
      under `atlases`.
- [ ] `schema: 1`, a positive `tileSize`, and only the allowed top-level keys.
- [ ] Sprite keys match the table above; every `rect` is inside its atlas image.
- [ ] Run `ATB_PACK=<dir> ./build-gui/tactical_battler`, hit *Fight*, confirm the
      `Loaded sprite pack …` console line and that undefined elements fall back
      cleanly.

See `ARCHITECTURE.md` §6 for the design rationale (why presentation is
authority-free) and `packs/README.md` for the in-directory quick reference.
